# Facade Reflection 0.6 Direct UI Membrane Visual Asset Index

## Purpose

Facade Reflection 0.6 records the current accepted visual asset family for the Section 1.2 Direct UI Membrane concept set.

This is a documentation-only asset index and acceptance matrix. It records surface roles, acceptance status, public-display safety, boundary risks, and required copy corrections for the current concept-stage anchors. It does not implement Section 1.2 and does not authorize runtime UI behavior, public gallery behavior, authentication, backend routes, database behavior, persistence, billing, public NEXUS execution, Palisade, Weave, customer data handling, production deployment, operational release authority, runtime monitoring, or compliance certification.

Section 1.2 remains not instantiated. No authenticated or interactive Direct UI Membrane exists yet.

The Access Boundary Membrane is public-display safe only when its non-authenticating framing remains prominent. The Workspace Control Surface is public-display safe after the AETHERUS spelling correction. The Escalation Review Surface is not public-display safe in its current form because "Escalation Active" can imply a live operational state.

## Required Corrections Before Broader Public Display

Before broader public display, apply these corrections to the visual assets or their adjacent copy:

1. Correct the AETHERUS spelling inconsistency on the Workspace Control Surface.
2. Replace "Escalation Active" on the Escalation Review Surface with "Escalation State Selected" or "Escalation Concept Selected."
3. Preserve non-authenticating framing on the Access Boundary Membrane so it does not imply working login, identity, tenant, or authorization behavior.
4. If "qualified reviewers" appears on the Evidence / Artifact Review Surface, soften it to "human review role" or "human review required."
5. Keep release language paired with review, eligibility, or concept framing.

These copy corrections keep the surface family in controlled review territory and away from operational-control claims.

## Boundary-Risk Notes

The main boundary risks are claim escalation through copy, status labels, and familiar interface patterns.

Access language can be mistaken for authentication if it looks like a working access-control system. Workspace language can be mistaken for an implemented authenticated surface if it suggests stored sessions, reviewer identity, tenant context, or collaboration. Evidence language can be mistaken for persistent audit infrastructure if it implies validated records, customer artifacts, immutable storage, or production audit retention. Release language can be mistaken for operational release authority unless it stays tied to review and eligibility. Escalation language can be mistaken for runtime monitoring or incident response if it uses active-state wording.

The accepted visual family should therefore remain evidence-first, lineage-aware, and explicitly concept-bound.
