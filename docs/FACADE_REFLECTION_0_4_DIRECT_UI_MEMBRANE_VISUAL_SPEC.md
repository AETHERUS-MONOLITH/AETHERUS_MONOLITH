# Facade Reflection 0.4 Direct UI Membrane Visual Specification

## Purpose

Facade Reflection 0.4 records the accepted visual and conceptual direction for Section 1.2 Direct UI Membrane.

This document is specification only. Section 1.2 Direct UI Membrane remains not instantiated. It is not implemented in AETHERUS_MONOLITH, not routed from the public site, not authenticated, not backed by a database, not persistent, and not connected to runtime monitoring, Palisade, Weave, NEXUS, customer data, or production release authority.

The accepted concept is a controlled human-review membrane. It is not a live AI operations dashboard.

## Canonical Surface Sequence

The accepted Section 1.2 surface sequence is:

1. Access Boundary Membrane
2. Workspace Control Surface
3. Evidence / Artifact Review Surface
4. Artifact Lineage Review
5. Release Review Chamber
6. Escalation Review Surface

The sequence should read as movement through review boundaries, not as movement through a live operations product.

## Canonical Anchors

Workspace Control Surface is the canonical Section 1.2 anchor and landing concept. It should orient the reviewer around the current concept review session, visible boundaries, available review chambers, and the evidence state that requires attention.

Evidence / Artifact Review Surface is the canonical evidence, lineage, and proof surface. It should prioritize artifact references, proof gaps, lineage context, and missing proof panels over status theatrics or operational metrics.

Release Review Chamber is the canonical release-state review surface. Release language must remain paired with review, eligibility, or conceptual framing. It must not imply operational release authority.

Escalation Review Surface remains accepted as the high-risk review-state surface. It should represent human review required, unresolved risk, blocked proof, and escalation state without incident-management or live-response language.

## Inherited Visual Grammar

Section 1.2 inherits its visual grammar from AETHERUS_MONOLITH:

- dark blue-black interface
- cyan structural linework
- amber/red/green state accents
- thin-line panels
- architectural geometry
- compressed uppercase labels
- institutional governance tone
- evidence-first hierarchy
- boundary strips and inspectors

This grammar should make the membrane feel like a controlled review instrument, not a marketing page, consumer dashboard, or simulated command center.

## Required Boundary Language

Any future visual or copy work for Section 1.2 should preserve explicit boundary language, including:

- Concept Mode
- No live AI execution
- No customer data
- No persistent audit ledger
- No runtime monitoring
- No production deployment
- No operational release authority
- No compliance certification
- No Palisade
- No Weave runtime

These statements describe the current concept boundary. They must not be softened into ambiguous claims of readiness.

## Accepted UI Patterns

The accepted visual direction may use the following patterns in future authorized design work:

- left navigation rail
- central review canvas
- right boundary inspector
- bottom boundary strip
- review chambers
- evidence chambers
- lineage thread
- missing proof panels
- reviewer notes as conceptual notes only

Reviewer notes must remain conceptual notes in this specification. They must not imply stored comments, authenticated reviewer identity, audit persistence, tenant-scoped collaboration, or database-backed records.

## Rejected UI Patterns

The following patterns are rejected for the current Section 1.2 concept boundary:

- telemetry
- charts implying live monitoring
- fake KPIs
- customer data
- production queues
- sync status
- export actions implying data infrastructure
- verified statuses implying validated records
- compliance badges
- certification language
- active incident-management language
- operational release buttons

The visual language should avoid anything that suggests live operations, customer deployment, runtime monitoring, certification, production queues, or real release authority.

## Copy Corrections to Carry Forward

Future visual work should carry forward these copy rules:

- Use "Mark Freeze State," "Mark Repair State," and "Mark Escalation State," not "Select" or "Approve."
- Use "Concept Review Session," not live session language.
- Use "Human review role" or "human review required," not unbounded "qualified reviewers."
- Keep "release" paired with "review," "eligibility," or "conceptual."
- Correct any AETHERUS spelling or logo inconsistencies in future visual work.

These corrections keep the membrane in review territory rather than operational control territory.
