# Facade Reflection 0.7 Direct UI Membrane IA Surface Taxonomy

## Purpose

Facade Reflection 0.7 formalizes the Section 1.2 Direct UI Membrane information architecture and conceptual surface taxonomy after the visual asset index recorded in Facade Reflection 0.6.

This document converts the accepted visual-concept set into a structured IA model: which conceptual surfaces exist, what each surface is for, how each surface relates to the others, what each surface may show, what each surface must not imply, which navigation/sidebar labels are canonical, which states are conceptual only, and which future implementation dependencies remain unresolved.

This is documentation-only. It does not implement Section 1.2 and does not authorize runtime UI behavior, public gallery behavior, authentication, backend routes, database behavior, persistence, billing, public NEXUS execution, Palisade, Weave, customer data handling, production deployment, operational release authority, runtime monitoring, or compliance certification.

## Current Object Status

Section 1.2 Direct UI Membrane remains not instantiated.

No authenticated or interactive Direct UI Membrane exists yet. The taxonomy below is a structural and specification artifact only. It describes a planned human review membrane as an IA model, not a working product surface.

AETHERUS_MONOLITH / Section 1.1 remains the instantiated static public artifact. It is the current static, browser-side, deterministic, scenario-driven, non-persistent, prototype-facing public Facade. Section 1.2 remains planned and specified only.

## Canonical Surface Taxonomy

The current official Section 1.2 conceptual set contains five surfaces:

1. Access Boundary Membrane
2. Workspace Control Surface
3. Evidence / Artifact Review Surface
4. Release Review Chamber
5. Escalation Review Surface

These surfaces should be understood as concept-stage review surfaces. Their relationship is directional but not routed: the Access Boundary Membrane frames entry into the planned membrane; the Workspace Control Surface is the canonical orientation anchor; Evidence Review, Release Review, and Escalation Review are focused review chambers that the workspace concept may reference or route reviewer attention toward in a future authorized implementation.

### Access Boundary Membrane

Function: entry / transition surface from the Section 1.1 public artifact into the planned Section 1.2 membrane.

Canonical labels:

- Access Boundary
- Concept Mode
- Governance Interface

May show:

- boundary language
- concept-stage entry framing
- non-authenticating transition copy
- visible reminders that Section 1.2 is not instantiated
- governance-interface framing that keeps the surface in review territory

Must not imply:

- real authentication
- real user accounts
- customer access
- login capability
- tenant identity
- authorization behavior
- production workspace access

### Workspace Control Surface

Function: canonical landing / orientation surface for the planned Direct UI Membrane.

Canonical role: routes reviewer attention across Evidence Review, Release Review, and Escalation Review.

May show:

- concept review session framing
- visible surface taxonomy
- review-state summaries
- boundary strips
- inspector panels
- links or labels for the conceptual review chambers
- evidence gaps that require human attention

Must not imply:

- live dashboard
- operational queue
- customer workspace
- runtime monitoring
- stored sessions
- authenticated reviewer identity
- tenant-scoped collaboration
- production control room

### Evidence / Artifact Review Surface

Function: evidence-first review and lineage/proof interpretation.

Canonical logic:

- supports
- does not support
- required proof
- review note

May show:

- artifact references
- lineage context
- proof gaps
- required proof panels
- reviewer-facing interpretation notes
- conceptual review notes
- boundary language explaining that notes are not persisted records

Must not imply:

- live evidence database
- persistent audit ledger
- immutable audit trail
- real-time logs
- validated customer artifacts
- production audit retention
- database-backed evidence records

### Release Review Chamber

Function: conceptual release-state review.

Canonical states:

- Freeze
- Repair
- Escalate

Canonical actions:

- Mark Freeze State
- Mark Repair State
- Mark Escalation State

May show:

- release review context
- release eligibility questions
- concept-stage state markings
- missing proof requirements
- human review notes
- explicit non-authority boundary language

Must not imply:

- actual release authority
- production approval
- deployment readiness
- operational release
- approved release
- live deployment gating
- certified production status

### Escalation Review Surface

Function: high-risk / boundary-conflict review-state surface.

Canonical language:

- Escalation State Selected
- Escalation Concept Selected

Prohibited language:

- Escalation Active

May show:

- high-risk review framing
- boundary conflict context
- unresolved proof requirements
- human review required language
- non-operational escalation-state selection
- repair or freeze context when relevant

Must not imply:

- live incident management
- enforcement
- alerts
- Palisade
- active production queue
- runtime monitoring
- operational incident response
- automated escalation handling

## Navigation / Sidebar Taxonomy

The canonical conceptual navigation labels are:

- Workspace
- Evidence Review
- Artifact Lineage
- Release Review
- Escalation Review
- Boundary

These labels are IA labels only. They are not implemented routes, not public links, not authenticated navigation, not sidebar behavior, and not evidence of a working Direct UI Membrane. Future routing, navigation behavior, sidebar state, keyboard access, search, filtering, and command palette behavior would require a separate authorized implementation pass.

## Information Hierarchy

Section 1.2.1 IA requirements should guide future concept work through the following hierarchy:

- role/task orientation first, so a human reviewer understands what kind of review surface they are conceptually entering
- critical review context before secondary detail, especially boundary conditions, missing proof, release review status, and escalation risk
- sidebar or collapsible navigation as a future concept only, not as implemented behavior in this pass
- search, filter, and command palette patterns as future implementation candidates only
- evidence-first hierarchy, with proof, lineage, required evidence, and claim support ahead of operational-looking metrics
- boundary strips and inspector panels for claim constraints, non-operational status, and unresolved dependencies

The intended hierarchy is review-first, evidence-first, and boundary-visible. It should not read as a metrics dashboard, customer workspace, production console, or live monitoring surface.

## State Language

Accepted conceptual state vocabulary:

- Concept Mode
- Conceptual Review Surface
- Non-operational
- Human Review Membrane
- Boundary Condition
- Required Proof
- Review Note
- Mark State

These terms keep the membrane in conceptual review territory. They may describe planned IA states, surface labels, copy anchors, and future design constraints, but they do not create product behavior.

Prohibited / risky state vocabulary:

- Active escalation
- Live queue
- Production-ready
- Operational dashboard
- Approved release
- Runtime monitor
- Certified
- Tenant-ready
- Customer workspace

These terms are prohibited or risky because they can imply operational authority, runtime behavior, customer deployment, certification, tenant infrastructure, or a live product surface that does not exist.

## Boundary Rules

The accepted concept set does not imply:

- live AI execution
- customer data
- persistent audit ledger
- runtime monitoring
- production deployment
- operational release authority
- compliance certification
- Palisade
- Weave runtime
- public NEXUS runtime
- authentication
- backend routes
- database behavior
- persistence
- billing

The concept set also does not imply tenant infrastructure, customer access, real user accounts, stored reviewer notes, immutable audit trails, live incident response, active alerts, production queues, public gallery behavior, or implemented Section 1.2 navigation.

Any future implementation, routing, authentication, data handling, runtime behavior, storage behavior, operational authority, Palisade instantiation, Weave runtime, or public NEXUS execution would require a separate authorized pass.
